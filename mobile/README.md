# Solarsearch mobile (Capacitor iOS shells)

Two native iOS shells that wrap the existing web apps so they can go to TestFlight /
the App Store, with native camera, geolocation and push. Each loads its hosted
**launcher page** over the air (`server.url`), so day‚Äëto‚Äëday changes ship by deploying
the website ‚Äî no App Store resubmission ‚Äî while the native bridge (camera/geo/push)
still works against the remote content.

> The `ios/` Xcode project is generated by `cap add ios` and **must be run on a Mac
> with Xcode**. It is not committed here.

## Two apps (one Capacitor project each)
| App | Folder | Bundle ID | Loads | Signed-in routing |
|-----|--------|-----------|-------|-------------------|
| **Field** (Sales Technician) | `mobile/field` | `net.solarme.solarsearch.tech` | `/app-field.html` | `sales_rep` ‚Üí `/tech.html` |
| **Installer** | `mobile/installer` | `net.solarme.solarsearch.installer` | `/app-installer.html` | `installer` ‚Üí `/installer.html` |

Live host: **https://www.solarsearch.com.au** (set in each `capacitor.config.ts`).

> **Confirm the bundle IDs** match the App IDs in your Apple Developer account before
> `cap add ios`. If you prefer a different namespace, change `appId` in the config first.

## Access + Apple requirements already wired in the web app
- **Access is by application** ‚Äî the launcher "Apply" button opens `/apply.html?role=‚Ä¶`,
  which creates the account if needed, records the **required T&C acknowledgement**, and
  writes `access_applications`. Admin grants in HQ ‚Üí Vetting ‚Üí "App access applications"
  (grant is blocked until T&Cs are accepted). On grant the role record is auto‚Äëprovisioned.
- **In‚Äëapp account deletion** (Apple 5.1.1(v)) ‚Äî a "Delete my account" action exists in
  `tech.html`, `install.html` and `apply.html` (`request_account_deletion`).
- **Privacy policy** ‚Äî `app-privacy.html`, live at `/app-privacy.html` (separate from the
  consumer `privacy.html`). Entity/ABN/region filled; confirm address/phone before submitting.
- **App Store metadata + privacy‚Äëlabel answers** ‚Äî `mobile/APP_STORE_METADATA.md`
  (copy‚Äëpaste into App Store Connect; includes the single‚Äërole demo‚Äëlogin note so review
  isn't blocked at the access gate).

## Build steps (on a Mac) ‚Äî per app
```bash
cd mobile/field        # or: cd mobile/installer
npm install
npx cap add ios        # generates ios/ (Mac only)
npx cap sync           # copies www + native plugins into the project
npx cap open ios       # Xcode ‚Üí set Team + bundle id ‚Üí app icon ‚Üí Product > Archive
```
Then in Xcode: select your Team (the Apple org you use for inspector/voya), confirm the
bundle id, add the app icon, and **Archive ‚Üí distribute to TestFlight**.

Because the app loads `server.url`, iterating on the web app needs **no rebuild** ‚Äî just
deploy the site. Rebuild only when you change native config, plugins, or the icon.

## Required ‚Äî iOS permission strings (Info.plist)
`cap add ios` creates `ios/App/App/Info.plist`. Add these (or the app is rejected / crashes
on first camera/GPS use) ‚Äî the text is shown in the OS permission prompt:
```xml
<key>NSCameraUsageDescription</key>
<string>Take site photos for your solar assessment.</string>
<key>NSLocationWhenInUseUsageDescription</key>
<string>Tag site photos and confirm on-site attendance.</string>
<key>NSPhotoLibraryUsageDescription</key>
<string>Attach drone or existing photos to a job.</string>
<key>NSPhotoLibraryAddUsageDescription</key>
<string>Save captured job photos.</string>
```

## App Store Connect checklist
